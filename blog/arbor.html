The Core Idea

Arbor is a paper about control, but its most interesting decision is representational. It does not ask the user to provide an image, a partial scan, a depth map, a set of keypoints, or a stronger prompt. Instead, it asks for a small typed 3D specification: this region should be filled, this region should remain empty, and this surface region should be touched. The generator is then asked to produce a plausible text-conditioned 3D asset that obeys those local geometric requirements without simply copying a target shape.

That distinction matters. A guide image or partial shape is usually treated as evidence of what the final object looks like. Arbor's constraints are closer to an artist's layout notes: "the object must pass through here," "do not occupy this volume," and "make contact with this support." In production terms, this is much closer to how people block out a scene or reserve functional space before details are known.

Core idea: Arbor turns 3D control into typed geometric conditioning. Constraint meshes are encoded as geometry tokens, routed to local parts of a frozen text-to-3D denoiser, and injected through trainable geometry adapters while the main generative prior stays fixed.

I find the paper worth reading because it treats negative space as a first-class control signal. Many controllable generation papers focus on what must appear. Arbor also cares about what must not appear. That sounds minor, but in 3D asset generation it is often the difference between a visually plausible object and an object that fits an actual design, collision, grasping, or layout constraint.

Problem Setting

The paper's goal can be written compactly as

\[ x \sim p(x \mid y, C), \]

where \(y\) is a text prompt, \(C\) is a set of typed 3D constraint meshes, and \(x\) is a generated 3D asset. The constraints are not a reconstruction target. They are partial requirements. A chair may need to touch a support plane and avoid a side volume; a toy may need to occupy a rough body envelope but leave a handle opening empty. The generator still has freedom over style, topology, details, proportions, and surface appearance.

This is difficult for three reasons. First, the control signal is sparse and local while the object prior is global. If the prompt says "a wooden stool" and a touch constraint asks for contact on a small patch, the system must satisfy the patch without hallucinating a large block just to make contact. Second, the constraints can conflict with the learned prior. Some prompts naturally prefer compact shapes; some constraints demand holes, protrusions, or support surfaces. Third, the control signal is 3D, but many strong generative 3D models are built around image-like conditioning or latent denoising pipelines that do not expose a clean slot for typed spatial rules.

The motivation is practical. Prompt-only text-to-3D is a slot machine when a user already knows part of the spatial intent. Image-conditioned generation gives stronger shape control, but it often freezes the design too early. Arbor targets the middle ground: keep the generative prior and variation, but let the user author spatial requirements in a native 3D form.

Background

TRELLIS as the Generative Prior

Arbor builds on the TRELLIS family of 3D generators. The relevant TRELLIS stage is the sparse-structure denoiser: a text-conditioned model that generates a sparse 3D latent grid. In Arbor's setting, the latent lattice is \(16^3\), and each lattice cell carries an 8-channel latent. Later stages refine and decode that structure into final 3D representations such as meshes, radiance fields, or 3D Gaussians.

This matters because Arbor does not retrain a full 3D generator. It attaches a trainable geometry-conditioning branch to a frozen text-to-3D denoiser. The pretrained model keeps its object prior; the new branch learns how to bias that prior toward local constraint satisfaction.

TRELLIS.2 as a Constraint Encoder

The constraint encoder comes from TRELLIS.2. TRELLIS.2 uses an object voxel representation that can encode both geometry and aligned per-surface attributes. Arbor repurposes that machinery. Instead of using the attribute channels only for material-like data, it stores surface normals plus three binary typed signals: hull, avoidance, and touch.

This is a sensible engineering choice. A generic image encoder would require rendering many views of the constraint object and would not naturally represent empty-space constraints. A 3D encoder preserves the spatial meaning of the control signal. It also lets avoidance regions remain volumetric rather than being compressed into an ambiguous 2D observation.

Control as Specification, Not Evidence

A partial scan says, "the final object should explain these measurements." A constraint mesh says, "the final object should satisfy these rules." Arbor belongs to the second category. That is why the three constraint types are important:

Hull

The object should occupy or support the indicated region. It is a positive geometric requirement, not necessarily a complete shape.

Avoidance

The generated surface should stay out of this volume. This encodes negative space, clearance, openings, and layout blockers.

Touch

The generated object should make contact with a specified region. This is useful for supports, attachments, handles, and functional interfaces.

The colors above mirror the paper's visual convention and carry actual semantic meaning. I avoid assigning different decorative colors to unrelated summary boxes later in the post because there is no extra category to encode there.

Method

Encoding Constraint Meshes

Arbor first merges the user-provided constraint meshes into a common 3D representation. The surface geometry, surface normals, and typed control signals are voxelized at high resolution, then passed through frozen TRELLIS.2 encoders. In the paper's notation, the two streams can be summarized as

\[ c_{\mathrm{shape}} = E_{\mathrm{T2,shape}}(\widetilde{C}_{\mathrm{shape}}), \qquad c_{\mathrm{signal}} = E_{\mathrm{T2,mat}}(\widetilde{C}_{\mathrm{normals}};\widetilde{C}_{\mathrm{signal}}). \]

Here \(\widetilde{C}_{\mathrm{shape}}\) is the voxelized constraint geometry, \(\widetilde{C}_{\mathrm{normals}}\) stores aligned normal channels, and \(\widetilde{C}_{\mathrm{signal}}\) stores the binary hull, avoidance, and touch flags. The semicolon denotes channel-wise concatenation. The resulting geometry memory is

\[ c_{\mathrm{geo}} = (c_{\mathrm{shape}};c_{\mathrm{signal}}). \]

The design is intentionally conservative. Shape and signal streams remain separate until the model has encoded them, which lets the adapter distinguish "there is geometry here" from "this geometry means keep-out" or "this geometry means touch." The ablations later show that this distinction is not cosmetic.

Why Routing Is Needed

A naive implementation would let every latent query attend to every geometry token. That is expensive and also semantically loose: a constraint near the chair leg should not dominate the denoising of a chair back unless global context demands it. Arbor instead partitions the \(16^3\) TRELLIS query lattice into 64 spatial groups, each covering a \(4\times4\times4\) block. For every group \(g\), the router creates a bounded context

\[ c_{\mathrm{ctx}}^{(g)} = [c_{\mathrm{local}}^{(g)};c_{\mathrm{global}}]. \]

The local part keeps the \(K=2048\) geometry tokens nearest to the group's center and corners:

\[ c_{\mathrm{local}}^{(g)} = \mathrm{TopK}_{2048} \left( \{(c_{\mathrm{geo}}^{(i)},p_i)\}_{i=1}^{N}, \min_{r\in R_g}\lVert p_i-r\rVert_2 \right). \]

The global part is a compact summary produced by learned summary tokens attending over all geometry tokens. In the paper's implementation this uses 96 global summary tokens. This local-plus-global split is the right inductive bias for constraints: most geometric rules are local, but an object still needs to remain globally coherent.

Injecting Geometry Into the Denoiser

Arbor inserts the geometry branch after the frozen text cross-attention and before the feed-forward layer in each TRELLIS block. Omitting residuals, normalization, and time modulation, the text path can be viewed as

\[ z_{\mathrm{text}}^{(\ell)} = \mathrm{CrossAttn}_{\mathrm{text}}^{(\ell)} \left( \mathrm{SelfAttn}^{(\ell)}(z^{(\ell-1)}), y \right). \]

Arbor then adds a geometry-conditioned residual:

\[ \Delta z_{\mathrm{geo}}^{(\ell,g)} = \mathrm{CrossAttn}_{\mathrm{geo}}^{(\ell)} \left( z_{\mathrm{text}}^{(\ell,g)}, c_{\mathrm{ctx}}^{(g)} \right), \]

\[ z^{(\ell,g)} = \mathrm{FFN}^{(\ell)} \left( z_{\mathrm{text}}^{(\ell,g)} + W_{\mathrm{geo}}^{(\ell)} \Delta z_{\mathrm{geo}}^{(\ell,g)} \right). \]

The projection \(W_{\mathrm{geo}}\) is zero-initialized. That is a small but important stabilizer: at the start of training, the adapter cannot destroy the pretrained denoiser's behavior. It must learn a useful residual geometry influence.

Training

Arbor trains only the geometry-facing modules: projection layers, geometry positional embeddings, global summary modules, semantic part token modules, and grounding adapters. The TRELLIS self-attention, text cross-attention, and feed-forward layers remain frozen. The training objective is the standard sparse-structure flow-matching objective used by the base model; the final Arbor run does not rely on a separate explicit compliance loss.

Training examples are built by taking 3D objects, generating or selecting typed constraints, encoding those constraints, and predicting the original sparse-structure latent. The paper reports voxelizing constraints at \(512^3\), encoding them into sparse \(32^3\) tokens, using 64 query groups, keeping 2048 local geometry tokens per group, and training the frozen-backbone checkpoint for about 80 hours on 8 GPUs with AdamW, EMA, fp16, and adaptive gradient clipping.

The data recipe is also pragmatic. The paper uses roughly 50k objects from ABO, HSSD, and an Objaverse-XL subset, then evaluates on Toys4K. That is much smaller than the data volume used by many foundation-style 3D generators, which makes the adapter framing more plausible: Arbor is not trying to learn all of 3D generation from scratch.

Experiments

Benchmarks and Metrics

Arbor evaluates on two constrained generation splits. The automatic split contains 128 procedurally generated constraints. The manual split contains 32 hand-authored constraints intended to be more out-of-distribution with respect to the training constraint generator. The baselines include the unconstrained TRELLIS generator, gradient-based constraint guidance, SpaceControl, and Spice-E, along with Arbor variants.

The metric design is unusually important here. If one only measures overlap with a positive shape, methods can overfill the guide and look artificially compliant. Arbor therefore reports separate terms:

  • Hull Hit: how well generated occupancy supports hull regions.
  • Avoid Viol.: occupied fraction inside forbidden avoidance volumes; lower is better.
  • Touch Hit: whether generated geometry reaches touch regions.
  • Volume Match: agreement with expected bounded occupancy, discouraging trivial overfilling.
  • MV-CLIP: multiview CLIP alignment with the text prompt.

The paper combines applicable terms using a harmonic-style control score:

\[ S_i = \{\mathrm{HullHit}_i,\mathrm{VolMatch}_i,1-\mathrm{AvoidViol}_i,\mathrm{MVCLIP}_i\}, \]

with \(\mathrm{TouchHit}_i\) appended when a touch constraint is present, and

\[ \mathrm{CtrlScr}_i = \frac{|S_i|} {\sum_{s\in S_i} 1 / \max(s,\epsilon)}. \]

The harmonic aggregation is appropriate: one bad term should hurt the score. A method that perfectly fills the hull but violates every avoidance region should not be called controlled generation.

The table says more than "Arbor wins." SpaceControl, for example, can score high on hull and avoidance terms but loses touch and volume balance. That is a typical failure mode of overly literal control: satisfy one geometric cue by sacrificing the object. TRELLIS remains surprisingly competitive in the user study because it produces clean, plausible assets, but it lacks direct spatial obedience. Arbor's value is the balance: stronger control without giving up too much of the base generator's visual prior.

This is one of the more useful evaluations in the paper. A controllable generator should not become a deterministic constraint-to-shape compiler. Arbor's variation score is high while its control score remains best among the displayed methods. That supports the paper's framing: constraints should restrict the sample space, not eliminate it.

Constraint Sweeps

The sweep examples are important because they test a different behavior from static compliance. If a method only memorizes a few constraint patterns, its outputs will jump or collapse as the guide changes. Arbor's qualitative sweeps suggest that the routed geometry branch produces a more graded response: the generated object adapts as the hull, touch, or keep-out region moves.

These ablations are the paper's strongest evidence for the architecture. The system does not work just because it sees a 3D blob. It needs local routing, shape features, and typed signals. The weak global-only result is especially useful: dense 3D control should not be treated like a single conditioning vector.

Strengths

Native 3D authoring interface

Hull, avoidance, and touch constraints match how artists and engineers specify spatial intent. The interface is more actionable than prompt-only control and less overdetermined than image conditioning.

Good use of frozen priors

Arbor keeps the base TRELLIS denoiser fixed and learns a geometry-facing attachment. This preserves the generative prior and makes the method more data-efficient than full retraining.

Routing fits the geometry

The local-plus-global router is a strong architectural choice. It gives nearby constraints high bandwidth while still exposing object-level context through summary tokens.

Evaluation catches tradeoffs

The paper reports control, volume, text alignment, variation, qualitative sweeps, ablations, and a user study. That makes it harder for a method to win by overfilling or collapsing diversity.

The visual treatment of these boxes is intentionally uniform. Unlike the hull/avoidance/touch cards, the four strengths are not distinct semantic signal types, so using multiple colors would add decoration without information.

Limitations

Main limitation: Arbor controls geometry at the sparse-structure stage. It does not yet provide the same direct control over later refinement, fine surface details, materials, or semantic function.

The first limitation is semantic. A touch region or hull region does not by itself say what functional role that region should play. If a constraint marks a seat-like support area, the model still has to infer "seatness" from geometry and prompt context. The paper's Arbor Semantics variant is an attempt in this direction, but it does not yet clearly outperform the routed geometry path.

The second limitation is stage coverage. Arbor operates on the sparse structure denoiser. Later TRELLIS refinement and decoding stages can soften or alter very local details. For many asset workflows, coarse geometric obedience is valuable, but contact edges, thin openings, and material boundaries may still require downstream cleanup or more direct control at later stages.

The third limitation is evaluation scope. The automatic and manual constraint splits are useful, but they are still authored or generated within the paper's experimental ecosystem. A broader constrained-3D benchmark would need more object categories, scene-level constraints, collision-heavy tasks, thin structures, physical support checks, and real user-authored production constraints. The paper's 64-resolution occupancy metrics also cannot fully capture delicate contact failures.

Finally, Arbor is tied to a particular backbone family. The design principle is broader than TRELLIS, but the implementation benefits from TRELLIS.2 encoders and a TRELLIS sparse-structure denoiser. Porting the idea to diffusion over meshes, Gaussian splats, implicit fields, or autoregressive 3D tokenizers would require new routing and injection choices.

Research Insight

Research insight: the paper's deeper contribution is not the three labels themselves. It is the idea that controllable generation should treat geometric constraints as typed specifications with both positive and negative spatial logic.

Many generative systems confuse conditioning with evidence. An image condition, partial point cloud, or rough mesh often says "make something like this." Arbor's constraints say something weaker but more useful: "whatever you make must obey these local rules." That is a better abstraction for design. It allows variation, but it also lets the user reserve space, enforce contact, and establish rough support without fixing the full object.

The router is also a useful design pattern. Dense 3D conditions should not be compressed into a global token if the desired behavior is spatially local. But pure locality is not enough either, because objects have global affordances and symmetries. Arbor's combination of nearby top-K geometry tokens and learned global summaries is a practical compromise that other controllable generation systems can reuse.

More broadly, this paper suggests that the next phase of 3D generative modeling will not be won only by better unconditional samples. Real value will come from control interfaces that match how people already think in 3D: block out space, constrain contact, reserve clearance, iterate, and then ask the model to fill in the uncertain parts.

Practical Perspective

For a research engineer, Arbor's implementation lesson is clear: keep the pretrained generator stable, add a narrow control branch, and make the control branch spatially routed. The zero-initialized geometry projection is not just a detail; it makes the adapter start as a no-op and learn a residual influence instead of perturbing the whole denoising process from the beginning.

For graphics and asset workflows, the interface is the most promising part. A modeler can create simple proxy meshes for support, clearance, and contact without sculpting the final asset. This is closer to constraint-based modeling than text prompting. It also suggests a natural UI: draw or import simple volumes, label them as hull/avoid/touch, generate several candidates, adjust constraints, and regenerate.

The method is not yet a push-button production solver. The outputs still need inspection, and thin contacts or functional semantics may fail. But Arbor moves the control handle to the right layer. It gives the user a geometric language instead of asking them to encode 3D spatial intent through prose.

Future Directions

  • Semantic constraint labels: extend hull, avoidance, and touch with part roles such as seat, handle, support, blade, grip, or connector.
  • Later-stage control: inject constraints into refinement and decoding stages so surface detail, material boundaries, and thin contacts remain controllable.
  • Standard constrained-3D benchmarks: evaluate on shared authoring tasks with collision checks, functional contacts, openings, supports, and scene-level layout constraints.
  • Interactive edit loops: support incremental constraint editing, partial regeneration, and local repair rather than one-shot generation from a fixed constraint set.
  • Multi-object and scene constraints: move from single assets to assemblies where contact, clearance, alignment, and support span multiple generated objects.
  • Backbone-agnostic adapters: test whether routed geometric conditioning transfers to mesh generators, Gaussian-splat generators, implicit-field diffusion, and autoregressive 3D token models.

Key Takeaways

  • Arbor makes negative space controllable. Avoidance regions are as central as positive hull constraints, which is a major practical improvement for 3D authoring.
  • The method is an adapter, not a full generator retraining recipe. It reuses frozen TRELLIS and TRELLIS.2 components while learning geometry-specific routing and injection modules.
  • Local routing is the core mechanism. The ablations show that global conditioning alone is far too weak for precise 3D control.
  • Good control should preserve variation. Arbor's fixed-constraint variation results support the idea that constraints should narrow the sample space without collapsing it.
  • The next frontier is semantics and refinement control. Constraint meshes encode where geometry should be, but not always what that geometry means or how fine detail should behave.
