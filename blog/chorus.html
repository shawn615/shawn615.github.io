The Core Idea

Chorus is a representation-learning paper for 3D Gaussian Splatting scenes. Its main claim is not that 3DGS renders well. That is already known. The more interesting claim is that optimized Gaussian primitives can be treated as a native 3D modality for pretraining a general scene encoder, if supervision is distilled carefully from complementary 2D foundation models.

The paper takes the "lift-then-align" idea from SceneSplat and makes it broader. SceneSplat primarily taught a 3DGS encoder language-aligned semantics. Chorus asks for a richer embedding: language alignment from SigLIP2, general visual structure from DINOv3, and object-aware spatial cues from PE-Spatial. A shared 3D encoder learns common factors, while lightweight teacher-specific heads absorb the incompatible parts of each teacher's feature space.

Core idea: render or lift 2D foundation-model features onto individual 3D Gaussians, train a feed-forward 3DGS encoder to predict those features, and use per-teacher projector heads so language, generic visual, and object-aware supervision can coexist without forcing all teachers into one brittle output space.

I read Chorus as part of a larger shift: 3DGS is becoming more than a rendering data structure. It is becoming a substrate for scene-level representation learning. The paper is strongest where it treats renderability as a supervision interface. Because Gaussians can be projected, alpha-composited, and re-rendered, they offer a clean bridge between 2D foundation models and feed-forward 3D encoders.

Problem Setting

The input is an optimized 3DGS scene, represented as a set of Gaussian primitives. Each primitive has a center, scale, orientation, opacity, and color. Chorus outputs per-Gaussian latent features through a shared encoder, with optional teacher-specific projections for language-aligned, DINO-style, or PE-Spatial-style features.

The downstream tasks are deliberately broad: zero-shot open-vocabulary semantic segmentation, open-vocabulary instance segmentation, linear and decoder probing, data-efficient supervised segmentation, point-cloud transfer, and 3D visual question answering through a GaussianVLM-style pipeline. This breadth matters. If a method only improves one segmentation benchmark, it may simply be a better semantic feature lifter. Chorus is trying to show that the learned 3D encoding is reusable.

The technically hard part is teacher conflict. SigLIP2, DINOv3, and PE-Spatial do not define the same representation space. One emphasizes language alignment, another general visual features, and the third object-aware spatial locality. Naively averaging or concatenating targets would likely produce a compromise embedding that is hard to optimize and hard to transfer. Chorus instead keeps a shared backbone and gives each teacher its own small projector and loss.

Background

3D Gaussian Splatting

A 3DGS scene stores many anisotropic Gaussian primitives rather than a mesh or a neural field. During rendering, each Gaussian is projected to the image plane and alpha-composited in depth order. This is useful for feature learning because the same rendering weights that explain RGB pixels can also relate 2D teacher features back to 3D primitives.

2D-to-3D Feature Lifting

Many open-vocabulary 3D methods use 2D foundation models as teachers. The usual recipe is to run a model such as CLIP, SigLIP, DINO, or SAM-related encoders on images, then transfer those 2D features to 3D points, voxels, Gaussians, or radiance-field samples through known camera geometry. Chorus follows this family, but it uses the lifted features as pretraining targets for a feed-forward 3DGS encoder rather than storing them only as scene-specific features.

Why Multi-Teacher Distillation Is Not Trivial

Multi-teacher distillation works only if the student can absorb complementary information without destructive interference. Language-aligned features are useful for open-vocabulary retrieval, but they may be spatially coarse. DINO-style features often carry strong part and correspondence structure, but they are not directly text-queryable. Object-aware features can improve grouping and locality, but they may overemphasize segmentation boundaries. Chorus makes this tension explicit through separate heads and balanced losses.

Method

3DGS Scene and Feature Lifting

The paper represents a 3DGS scene as

\[ \mathcal{G} = \{(\mathbf{x}_i,\mathbf{s}_i,\mathbf{q}_i,\alpha_i,\mathbf{c}_i)\}_{i=1}^{N}, \]

where \(\mathbf{x}_i\) is the center, \(\mathbf{s}_i\) the scale, \(\mathbf{q}_i\) the orientation quaternion, \(\alpha_i\) the opacity, and \(\mathbf{c}_i\) the color of Gaussian \(i\). For a pixel \(\mathbf{u}\) from viewpoint \(p\), rendering follows alpha compositing:

\[ \mathbf{C}(\mathbf{u}\mid p) = \sum_{i\in\mathcal{S}_{d,\mathbf{u}}} T_i\,\alpha_i(\mathbf{u}\mid p)\,\mathbf{c}_i, \qquad T_i=\prod_{j<i}\left(1-\alpha_j(\mathbf{u}\mid p)\right). \]

The same weights can lift a 2D teacher feature \(F_{p,\mathbf{u}}\) into a per-Gaussian target feature. If \(\mathcal{S}_i\) is the set of view-pixel pairs where Gaussian \(i\) contributes, then Chorus uses a normalized weighted average:

\[ f_i = \sum_{(p,\mathbf{u})\in\mathcal{S}_i} \bar{w}_i(p,\mathbf{u})F_{p,\mathbf{u}}, \qquad \bar{w}_i(p,\mathbf{u}) = \frac{w_i(p,\mathbf{u})} {\sum_{(p',\mathbf{u}')\in\mathcal{S}_i} w_i(p',\mathbf{u}')}. \]

This is the key geometric bridge. It converts dense 2D teacher maps into per-Gaussian pseudo-labels using the exact visibility and compositing behavior of the 3DGS scene.

Shared Encoder, Teacher-Specific Projectors

Chorus uses a shared 3D encoder \(g_\theta\) that maps the Gaussian set into per-Gaussian latent features:

\[ Z = g_\theta(\mathcal{G}) \in \mathbb{R}^{N\times d_z}. \]

Each teacher \(t\in\mathcal{T}=\{\text{lang},\text{dino},\text{pe}\}\) has its own projector \(h_t\), implemented as a two-layer MLP with LayerNorm and GELU:

\[ \hat{f}^{(t)} = h_t(Z) \in \mathbb{R}^{N\times d_t}. \]

This design is conservative but important. A single projector would force SigLIP2, DINOv3, and PE-Spatial to share an output coordinate system. Chorus instead uses one backbone for reusable scene structure and small heads for teacher-specific feature geometry.

Matching Loss and Total Objective

For each teacher, the base matching loss combines angular and magnitude agreement:

\[ \mathcal{L}_{\mathrm{match}}^{(t)} = \frac{1}{|M^{(t)}|} \sum_{i\in M^{(t)}} \lambda_1 \left(1-\cos(\hat{f}^{(t)}_i,\widetilde{f}^{(t)}_i)\right) + \lambda_2 \mathrm{SmoothL1}(\hat{f}^{(t)}_i,\widetilde{f}^{(t)}_i). \]

The teacher features are standardized using PHI-S, which applies a PCA rotation and isotropic Hadamard scaling so that teacher feature scales are easier to balance. When semantic or instance labels are available, Chorus also uses teacher-specific contrastive terms: semantic class pooling for the language teacher and instance pooling for PE-Spatial.

With an active teacher set \(\mathcal{A}(e)\) at epoch \(e\), the total objective is

\[ \mathcal{L}_{\mathrm{total}}(e) = \sum_{t\in\mathcal{A}(e)} \lambda_t \left( \mathcal{L}_{\mathrm{match}}^{(t)} + \eta_t\mathcal{L}_{\mathrm{con}}^{(t)} \right). \]

The implementation uses equal teacher weights \(\lambda_t=1.0\) and contrastive weight \(\eta_t=0.02\). PE-Spatial is introduced in a staged way rather than from the first epoch, which the ablations suggest is better than forcing object-aware local supervision too early.

The important separation is conceptual: teacher heads absorb teacher-specific feature geometry, while the shared encoder learns the reusable per-Gaussian scene representation.

Render-and-Distill Adaptation

The main pretraining pipeline uses offline lifted pseudo-labels. For new domains, Chorus introduces a cheaper adaptation loop. It renders RGB views from the 3DGS, runs the 2D teachers online, renders the encoder's predicted feature maps using Gaussian compositing, and matches the rendered predicted features to teacher features in image space.

The predicted feature map for teacher \(t\) is

\[ \hat{F}^{(t)}_{p,\mathbf{u}} = \sum_{i\in\mathcal{S}_{p,\mathbf{u}}} w_i(p,\mathbf{u})\hat{f}^{(t)}_i. \]

The image-space adaptation loss applies the same feature matching over valid pixels:

\[ \mathcal{L}_{\mathrm{img}}^{(t)} = \frac{1}{|\Omega|} \sum_{(p,\mathbf{u})\in\Omega} \ell_{\mathrm{match}} \left( \hat{F}^{(t)}_{p,\mathbf{u}}, \widetilde{F}^{(t)}_{p,\mathbf{u}} \right). \]

This is a useful engineering contribution because it avoids precomputing and storing all 3D teacher labels for a new dataset. On InteriorGS, the paper reports 1080 GB of storage for the uplifting route versus 8 GB for rendering-based adaptation, with less than 0.1 seconds per view of rasterization overhead during training.

Training and Data

The pretraining backbone adapts the five-stage transformer encoder from Sonata and uses a 512-dimensional bottleneck feature. The standard Chorus model consumes all 3DGS parameters: center, color, opacity, quaternion, and scale. A point-cloud variant uses only Gaussian centers, colors, and estimated normals, leaving the teacher targets and losses otherwise unchanged.

The pretraining scenes come from SceneSplat-7K, with newly processed pseudo-labels for SigLIP2, DINOv3, and PE-Spatial. For DINOv3 and PE-Spatial, the paper reports 1024-dimensional per-Gaussian pseudo-labels. SigLIP2 provides 1152-dimensional features. The authors also reprocess the Matterport3D subset because the original region-level 3DGS quality had limited multi-view coverage.

InteriorGS is used as a new 3DGS-native benchmark rather than as pretraining data. The paper maps its original 755 fine-grained labels into 72 classes, assigns semantic and instance labels to Gaussians with a connected-components heuristic inside bounding boxes, removes outliers, and uses scenes 001-100 as test, 101-200 as validation, and 201-1000 as training for downstream experiments and adaptation.

Experiments

Zero-Shot 3D Semantic Segmentation

The central 3DGS-native result is zero-shot semantic segmentation. With joint training on ScanNet, ScanNet++, and Matterport3D, Chorus improves over SceneSplat on all four evaluation datasets in the selected table below. The largest practical gain is on InteriorGS, which is a new-data generalization test: f-mIoU moves from 10.0 to 15.7, and f-mAcc from 18.3 to 24.1.

Open-Vocabulary Instance Segmentation

Instance segmentation is a more demanding test because semantic alignment alone is not enough. The model needs grouping and object boundaries. Chorus uses the same Mask3D proposals as the 3D-only baselines and improves over Mosaic3D on mAP25 and mAP50. The tail-class gain is especially notable: 13.0 mAP on tail classes versus 5.4 for Mosaic3D, while head-class mAP is lower than Mosaic3D.

Point-Cloud Transfer

One of the more surprising results is that the point-cloud variant of Chorus transfers well even though the supervision was built from 3DGS scenes. It uses centers, colors, and estimated normals, and it is compared against Sonata and other point-cloud pretraining methods. The paper argues that 3DGS pretraining acts partly like a strong augmentation source, while multi-teacher supervision supplies semantically useful scaling signals.

Render-and-Distill Adaptation

The adaptation result is less glamorous than the main leaderboard numbers, but it is practically important. Instead of building massive per-Gaussian pseudo-label stores for every new domain, the model can render views and distill online. On InteriorGS, 100 additional scenes give a reported +2.7 mIoU gain under linear probing over the standard pretrained model.

3D Question Answering

Chorus is also plugged into GaussianVLM by replacing the SceneSplat 3D backbone and feeding only the final Chorus encoder stage. The result is not a complete new VQA architecture, but the controlled backbone swap is still informative: ScanQA exact match improves from 14.4 to 14.8, ROUGE from 34.8 to 37.4, and Nr3D ROUGE from 19.2 to 28.8, while ScanQA METEOR drops slightly from 22.9 to 22.5.

Ablations

The teacher ablation is one of the most important pieces of evidence because it tests the paper's central design claim. On ScanNet++ pretraining, adding DINO to language supervision improves foreground mIoU on the validation split from 27.1 to 29.4, and adding PE-Spatial nudges it to 29.6. On InteriorGS, the same sequence improves f-mIoU from 8.0 to 9.3 to 11.4. The gain is not enormous in every column, but the direction supports the complementarity argument.

The design-choice ablation also matters. SmoothL1, 3DGS-aware augmentations, staged PE-Spatial introduction, and instance-level contrastive learning each contribute incremental gains. This is a good sign: the final result is not just one giant teacher model doing all the work. There are representation and optimization choices that measurably affect the learned 3D encoder.

Strengths

Representation-first framing

The paper treats 3DGS as a native modality for pretraining, not just a target for rendered image features.

Clean teacher separation

Shared backbone plus per-teacher projectors is a pragmatic way to avoid forcing incompatible teacher spaces together.

Broad evaluation

The paper tests open-vocabulary semantics, instances, probing, low-data finetuning, point clouds, and VQA rather than a single metric.

Adaptation story

Render-and-distill adaptation is directly tied to 3DGS renderability and reduces the cost of moving to a new domain.

Limitations

The first limitation is cost hidden outside the final encoder. Chorus reduces downstream inference to a feed-forward 3D model, but pretraining still depends on running large 2D teachers and lifting their features. The conclusion explicitly notes the offline cost of precomputing teacher pseudo-labels as a remaining limitation.

The second limitation is teacher bias. Chorus inherits the strengths and blind spots of SigLIP2, DINOv3, and PE-Spatial. If the 2D teachers fail on transparent objects, unusual room layouts, reflective materials, or long-tail categories, the 3D encoder can only partially correct that through geometry and multi-view aggregation.

The third limitation is modality dependence. The point-cloud transfer result is impressive, but the core method is still built around 3DGS scenes and their rendering weights. For raw point clouds without a 3DGS source or without reliable color/normal estimates, the direct route is less clear.

Finally, the VQA experiment is best read as a backbone-swap study rather than proof of a new reasoning system. Chorus improves several metrics, especially Nr3D ROUGE, but the full reasoning stack still depends on GaussianVLM, OPT-1.3B, LoRA finetuning, and the task-specific sparsifier.

Research Insight

Research insight: renderability is a representation-learning interface. 3DGS does not merely store scene appearance; it provides differentiable, visibility-aware routes for moving supervision between 2D foundation models and 3D primitives.

The main design principle is worth generalizing: use the geometric representation to define how supervision flows, then use a neural encoder to amortize that supervision across scenes. Chorus lifts teacher features into 3D once, trains a feed-forward encoder, and later avoids re-running expensive 2D teachers at inference.

The paper also highlights a useful compromise in multi-modal distillation. Instead of learning one universal feature vector that must simultaneously be a language embedding, a DINO embedding, and an object-aware embedding, Chorus learns a shared latent representation that can be projected into each teacher's space. That is a more realistic hypothesis about representation convergence: the common factors can be shared, while teacher-specific coordinate systems remain separate.

For 3D foundation models, this suggests that the next useful scaling unit may not be a larger point-cloud dataset alone. It may be a large collection of renderable 3D scenes where 2D supervision can be queried, projected, standardized, and distilled. 3DGS is attractive here because it gives geometry, appearance, fast rendering, and per-primitive structure in one package.

Practical Perspective

For a research engineer, Chorus is most useful as a recipe for building reusable 3D scene features. If you already have optimized 3DGS scenes, the method gives a concrete path: collect teacher feature maps, lift them to Gaussians, train a sparse or point-style transformer encoder, and keep teacher-specific heads during pretraining.

For embodied AI and XR systems, the practical value is amortization. A deployed system cannot afford to run several 2D foundation models from many views every time it needs semantic features. A feed-forward 3DGS encoder that has already absorbed those teacher signals is much more compatible with interactive scene understanding.

The runtime is not free. The reported model has 131.3M parameters and takes 0.70 seconds per InteriorGS scene on average, compared with 0.65 seconds for SceneSplat and 0.49 seconds for Sonata in the paper's runtime table. Still, for scene-level processing over nearly one million Gaussians per scene, that is a practical regime.

Future Directions

  • Unified 3DGS and point-cloud encoder: the authors explicitly mention this as a promising next step. The current point-cloud result suggests shared supervision can transfer, but the architectures are still separated by input modality.
  • Online or cached teacher distillation: reduce the offline pseudo-labeling cost without losing the stability of uplifted supervision.
  • More teacher diversity: test depth, normal, affordance, correspondence, and temporal teachers, while preserving the projector separation that prevents objective conflict.
  • Dynamic scenes: extend the scene encoder to time-varying Gaussian splats, where instance identity and feature consistency across time become central.
  • Robustness benchmarks: evaluate under noisy captures, non-MCMC optimized splats, sparse views, sensor artifacts, and outdoor scenes where 3DGS quality varies strongly.
  • Better VLM integration: train language models that consume Chorus features more natively instead of only swapping the encoder inside an existing GaussianVLM pipeline.

Key Takeaways

  • Chorus turns 3DGS into a pretraining modality. It learns per-Gaussian scene features directly from optimized splat parameters.
  • The multi-teacher design is the main contribution. SigLIP2, DINOv3, and PE-Spatial supervise one shared encoder through separate projector heads.
  • The evaluation is broad enough to be meaningful. The paper covers zero-shot semantics, instances, probing, low-data finetuning, point-cloud transfer, and VQA.
  • Render-and-distill adaptation is a practical addition. It exploits 3DGS renderability to avoid large 3D pseudo-label stores for new domains.
  • The main open problem is cost and generality. Precomputing teacher supervision remains expensive, and the method is still most natural when renderable 3DGS scenes are available.
